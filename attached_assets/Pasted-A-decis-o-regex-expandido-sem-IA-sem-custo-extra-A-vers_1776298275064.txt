A decisão: regex expandido — sem IA, sem custo extra
A versão híbrida (com GPT-4o-mini como fallback) seria mais robusta a longo prazo, mas tem dois problemas práticos para agora: adiciona latência em todo request que cai em other, e cria uma dependência externa que pode falhar. O intent-detector é chamado antes de qualquer coisa — se ele travar, trava tudo.

A análise dos testes existentes mostra que os buracos do regex atual são completamente previsíveis — são variações naturais do português que qualquer pessoa usaria. Não é necessária IA para cobrir esses casos. Regex expandido resolve 95% dos falsos negativos sem adicionar nenhum risco ou custo.

Os casos que estão caindo em other hoje
scheduling — faltando
"reservar uma consulta"
"marcar hora"
"quero ir ao dentista"
"tem espaço na agenda"
"quero um encaixe"
"fazer uma consulta"
"preciso de atendimento"
"posso marcar pra hoje"
"quero ir hoje"
"agendar horário" (com acento)
outros — faltando
"nao consigo comparecer" → cancel
"nao da pra ir" → cancel
"encaixar em outro dia" → reschedule
"quanto fica o implante" → price
"tem previsao de valor" → price
"voces parcelam" → price
"nao tenho condicao" → objection
"to com medo" → objection
"oi, boa tarde!" → greeting
"eai" → greeting
Quando scheduling vira other, o shouldSkipScheduleOffer fica true e a agenda nunca é carregada. A IA responde sem saber os horários disponíveis — daí o comportamento estranho de "não oferecer horário" mesmo quando o lead claramente quer marcar.

O que mudar no arquivo intent-detector.ts

Passo 2 — Adicionar normalização de acentos antes do match
Adicionar um passo de normalização antes de testar os padrões. Isso elimina a necessidade de colocar variações com/sem acento em cada regex:

function normalize(text: string): string {
  return text
    .normalize("NFD")
    .replace(/[\u0300-\u036f]/g, "") // remove acentos
    .toLowerCase()
    .trim();
}

export async function detectIntent(message: string): Promise {
  const normalized = normalize(message);

  for (const [intent, pattern] of INTENT_PATTERNS) {
    if (pattern.test(normalized)) {
      return intent;
    }
  }

  return "other";
}
Com isso, "agendá-lo", "horário", "convênio" e "não" passam a bater nos padrões sem precisar de variantes duplicadas. As regexes ficam mais curtas e fáceis de manter.

Quando considerar a versão híbrida no futuro
A versão híbrida (regex → se other → GPT-4o-mini) faz sentido quando o sistema estiver estável e você quiser cobrir mensagens imprevisíveis que regex nunca vai pegar — metáforas, gírias novas, frases muito indiretas ("minha filha tá com dor, o que faço?"). O custo seria de ~50 tokens por chamada que cai em other, com cache de resultado por 60s. Mas para agora, o regex expandido é a escolha certa: zero latência extra, zero custo, zero risco de falha externa.
